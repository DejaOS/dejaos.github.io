## 07-20

### Easier Device Boot Background Replacement

Previously, replacing the **device** boot background (the image shown on power-on / reboot) usually meant rebuilding or reflashing firmware, which was inconvenient and often required vendor assistance.

After further research, we now provide a simpler approach: a **device-specific DejaOS upgrade app**. Add your custom image into the app we provide, then run it on the device to update the boot background—no full firmware flash required in typical cases.

Notes:

1. **Different device models use different upgrade apps.** We supply the matching package for your model. This capability will later be integrated into the [System Management App](/docs/basics/managerapp); it is **not** published on GitHub for now.
2. **Image resolution must match the device screen.** You can look up the resolution for your model in `dxDriver.js`.
3. **Factory pre-install** is not the default for small orders. For **large-volume** devices we can offer a pre-install service; otherwise an extra fee may apply—contact the DejaOS team.

(Application startup UI backgrounds remain ordinary `dxui.Image` resources and can still be replaced by updating project assets.)

## 06-12

### DejaOS CLI Initial Release

The standalone **DejaOS CLI** command-line development tool is now available. Developers can install components, build projects, connect USB devices, perform incremental or full code synchronization, control applications, reboot devices, and stream real-time logs without installing the VSCode extension.

- Supports commands including `dejaos install`, `dejaos build`, `dejaos sync`, `dejaos sync --all`, `dejaos run`, and `dejaos logs`;
- Provides the `dejaos edit` terminal project configuration editor for managing device models, SDK versions, and component dependencies;
- Supports global installation through npm: `npm install -g dejaos-cli`;
- Designed for terminal development, automation scripts, and AI-assisted programming workflows.

For detailed usage, see the [DejaOS CLI](/docs/basics/cli) documentation.

## 04-13

### Introduced SDK Versioning (2.0 / 3.0)

To support the video intercom refactor, we re-architected the camera-centric pipeline and clearly separated components such as `capture`, `ivcore`, `display`, `facial`, and `intercom`, so the same camera stream can be used concurrently for multiple scenarios including intercom, display, and recognition. Because the new and legacy component stacks differ at both architecture and interface levels, they are no longer backward compatible.  
To make version management and project selection clearer, we introduced the **SDK version** concept: **SDK 2.0** includes all versions in the legacy component stack, while **SDK 3.0** includes all versions in the new component stack.

## 04-02

### Video Intercom Support for Face Recognition Devices

Face recognition devices now support video intercom, enabling visitors to initiate a door call and allowing users to answer from indoor units or mobile apps with real-time video and two-way audio. After confirming the visitor identity during the call, users can complete remote door release as part of the same workflow, with support for private server deployment.  
For detailed architecture and call flow, refer to the [Video Intercom System Overview](https://dejaos.com/docs/videointercom/overview/) documentation.
